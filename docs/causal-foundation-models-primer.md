# Causal Foundation Models: Wiki + Primer

Last researched: 2026-05-28

This note is a working map for exploring foundation models for tabular,
relational, time-series, and enterprise data where the central question is
causal: "What would happen if we intervened?"

## 1. One-page Summary

A causal foundation model is not just a large predictive model. It is a
pretrained, reusable model that can adapt to many causal tasks across datasets,
schemas, and domains while making the causal estimand and assumptions explicit.

The clearest current path is based on prior-data fitted networks (PFNs):
transformers are pretrained on many synthetic data-generating processes, often
structural causal models (SCMs), and then use in-context learning to answer a new
causal query from a small dataset without task-specific training.

The field is early. There is no single general-purpose causal foundation model
that can safely infer arbitrary causes from arbitrary enterprise data. Current
work is promising for bounded tasks such as average treatment effect (ATE),
conditional average treatment effect (CATE), causal effect estimation under
specific assumptions, causal discovery over small graphs, and time-series causal
effect estimation with simulated priors.

The core research tension:

- Foundation models compress broad inductive bias and amortize repeated work.
- Causal inference depends on identification assumptions that data alone often
  cannot verify.
- A useful causal foundation model should automate estimation and sensitivity
  analysis, but it should not hide the assumptions that make the answer causal.

## 2. Why This Matters

Enterprise data science often starts as prediction:

- Which customers will churn?
- Which machine will fail?
- Which transaction is risky?
- Which lead will convert?

Causal decision-making asks a different question:

- What happens if we lower price by 5%?
- What happens if we change a credit policy?
- What happens if sales outreach targets this segment?
- What happens if a manufacturing parameter is adjusted?
- Which patients benefit from a treatment, not merely which patients have high
  risk?

Prediction can optimize observation. Causality supports intervention. A causal
foundation model would be valuable if it could make causal workflows faster,
more repeatable, and more robust across the many structured datasets that drive
business and scientific decisions.

## 3. Minimal Glossary

**Foundation model**

A model pretrained on broad data or tasks that can be adapted to many downstream
tasks. In tabular and causal settings, "broad data" can mean millions of
synthetic datasets, real datasets, relational databases, or simulated causal
systems.

**Structural causal model (SCM)**

A model of the data-generating process. Variables are generated by mechanisms:
each variable is a function of its direct causes and exogenous noise.

**Causal graph / DAG**

A directed graph where arrows encode direct causal relationships. A graph is not
just a correlation map; it represents assumptions about how the world generates
data.

**Intervention**

An action that sets or changes a variable, often written as `do(X=x)`. This is
different from conditioning on `X=x`; conditioning observes a subset of the data,
while intervening changes the data-generating process.

**Estimand**

The causal quantity being targeted. Examples: ATE, CATE, odds ratio under an
intervention, direct effect, indirect effect, policy value.

**ATE**

Average treatment effect: the average difference in outcome if everyone received
treatment versus if everyone did not.

**CATE**

Conditional average treatment effect: the treatment effect for a subgroup or
context, such as a customer segment, patient profile, or plant operating regime.

**Confounder**

A variable that causally affects both treatment and outcome. Failing to adjust
for confounders can make treatment effects biased.

**Collider**

A variable caused by two other variables. Conditioning on a collider can create
spurious associations.

**Back-door adjustment**

A method for identifying causal effects by adjusting for variables that block
non-causal paths between treatment and outcome.

**Front-door adjustment**

A method that can identify effects through a mediator under specific assumptions,
even when there is unobserved confounding between treatment and outcome.

**Instrumental variable**

A variable that affects the treatment, affects the outcome only through the
treatment, and is independent of unobserved confounders. Instruments are powerful
but hard to justify.

**Counterfactual**

A question about what would have happened to the same unit under a different
intervention.

**Causal discovery**

Learning a causal graph or graph equivalence class from data plus assumptions.
Observational data alone is usually insufficient without additional assumptions,
interventions, time order, invariances, or prior knowledge.

**Prior-data fitted network (PFN)**

A neural model pretrained on datasets sampled from a prior over data-generating
processes. At inference time, it takes a new dataset as context and performs a
task in one or a few forward passes.

**In-context learning (ICL) for tables**

Supplying training rows, labels, query rows, and sometimes task metadata directly
as model context. The model adapts through the context rather than through weight
updates.

**Amortized inference**

Paying a large training cost once so that inference on future datasets is fast.
This is central to the PFN view of causal foundation models.

## 4. What Would Count as a Causal Foundation Model?

A practical definition:

> A causal foundation model is a pretrained model that accepts data plus a causal
> query and returns an identified causal answer, uncertainty, or diagnostic under
> explicit assumptions, across many tasks or domains.

Expected inputs:

- Data: tabular rows, relational tables, time series, event logs, experiments,
  observational records, or mixtures of these.
- Query: treatment, outcome, unit of analysis, time horizon, target population,
  and estimand.
- Assumptions: known graph, partial graph, forbidden edges, time ordering,
  randomized assignment, instruments, mediators, negative controls, or domain
  constraints.
- Context: metadata, schema, feature semantics, previous experiments, domain
  priors, or source-target population information.

Expected outputs:

- Effect estimates: ATE, CATE, ITE proxies, dynamic effects, mediation effects.
- Uncertainty: credible intervals, confidence intervals, posterior samples, or
  calibrated prediction bands.
- Diagnostics: overlap, positivity, balance, sensitivity to hidden confounding,
  placebo tests, negative controls, and robustness checks.
- Graph beliefs: posterior over graphs, equivalence classes, or recommended
  adjustment sets.
- Decision support: policy value, uplift ranking, expected utility, and
  counterfactual simulation.

Important constraint:

A causal foundation model should not claim "causality from data alone" as a
general rule. It should surface what assumptions turn an observed pattern into a
causal claim.

## 5. Causal Ladder for Foundation Models

A useful capability ladder:

1. Predictive tabular foundation model

   Learns `P(Y | X)` across datasets. Useful, but not causal by itself.

2. Causal-assumption-aware predictor

   Uses causal priors, invariance, or graph constraints to improve robustness and
   transfer.

3. Causal effect estimator

   Estimates ATE/CATE under a known or assumed identification strategy.

4. Causal discovery assistant

   Produces graph hypotheses, uncertainty, and testable implications from data
   plus constraints.

5. Counterfactual model

   Supports unit-level "what if this unit had received another intervention?"
   queries under a structural model.

6. Policy and intervention planner

   Chooses interventions under constraints, costs, fairness requirements, and
   distribution shift.

7. Relational and temporal causal world model

   Handles multi-table schemas, time-varying treatments, feedback loops,
   interference, delayed effects, and changing environments.

## 6. Current Research Landscape

### 6.1 Foundations: Causal Inference and Causal Representation Learning

Classical causal inference gives the language of interventions, counterfactuals,
identification, adjustment, and sensitivity analysis. The key lesson is that
estimation quality is not enough: the estimand must be identifiable under the
assumptions.

Causal representation learning extends this to modern ML: if data arrives as raw
images, text, sensors, or embeddings, the causal variables may not be directly
given. A model may need to discover high-level variables that correspond to
stable causal mechanisms.

Relevance for this project:

- Enterprise data already has named columns, tables, and events, but those names
  are imperfect proxies for causal variables.
- Causal foundation models need both statistical estimation and representation
  of mechanisms, entities, time, and interventions.

### 6.2 Tabular Foundation Models

Tabular foundation models are the closest engineering precedent.

TabPFN showed that a transformer can be pretrained on synthetic tabular tasks and
then solve new small tabular prediction problems with in-context learning. TabPFN
v2, published in Nature in 2025, substantially strengthened this line and framed
TabPFN as a tabular foundation model. Later reports and models extend scale,
relational support, tabular-text support, time-series integration, and faster
explanations.

Other tabular foundation model directions include TabDPT, TabICL, TabSTAR, CARTE,
and real-data continued pretraining.

Relevance for this project:

- Tables are the dominant enterprise substrate.
- PFN-style pretraining can encode priors over data-generating processes.
- But predictive excellence does not imply causal validity.

### 6.3 Relational Foundation Models and Relational Deep Learning

Most enterprise data is relational: customers, orders, products, providers,
locations, claims, assets, tickets, and events live across multiple linked
tables. Flattening a relational database into one feature table loses structure
and can introduce leakage.

RelBench and relational deep learning show a path away from manual feature
engineering toward models that learn from primary-foreign key structure. This is
important for causal foundation models because many interventions are relational:
pricing a product affects customers; supply changes affect downstream orders;
network policies affect connected users.

Relevance for this project:

- A realistic causal foundation model should understand schemas, entities,
  foreign keys, temporal joins, and unit-of-analysis boundaries.
- Interference is common: one unit's treatment can affect another unit's outcome.

### 6.4 PFN-based Causal Foundation Models

This is the most direct current causal foundation model line.

The basic recipe:

1. Define a prior over SCMs or causal data-generating processes.
2. Simulate many datasets with known causal targets.
3. Train a transformer/PFN to map observed data and a query to the target causal
   quantity.
4. At test time, pass a new dataset as context and return an effect estimate or
   posterior without fitting a new model from scratch.

Representative directions:

- CInA connects attention to covariate balancing and targets zero-shot treatment
  effect estimation.
- Do-PFN trains on synthetic causal structures, including interventions, to
  estimate interventional outcomes from observational data.
- CausalPFN amortizes causal effect estimation under ignorability and targets
  ATE/CATE workflows.
- CausalFM gives a broader PFN framework for back-door, front-door, instrumental
  variable, and related settings through causal Bayesian priors.
- CausalTimePrior extends the PFN idea to temporal SCMs with observational and
  interventional time series.
- Recent work studies consistency, calibration, prior-induced confounding bias,
  real-world simulation stress tests, causal sensitivity analysis, and amortized
  causal discovery.

Relevance for this project:

- This is likely the first serious prototype path.
- The main design object is the synthetic causal prior.
- Evaluation must test prior mismatch, unobserved confounding, calibration, and
  out-of-distribution causal structures.

### 6.5 Classical Causal ML Tooling

Libraries such as DoWhy, EconML, CausalML, and Ananke are not foundation models,
but they encode mature workflows:

- define a causal model,
- identify an estimand,
- estimate the effect,
- refute or stress-test the estimate.

These tools are useful baselines and scaffolding. A causal foundation model
should ideally interoperate with this workflow rather than replace it with an
opaque "answer machine."

## 7. Architectural Sketch

A causal foundation model for enterprise data can be thought of as five
components.

### 7.1 Data and Schema Encoder

Handles:

- numeric, categorical, text, timestamp, and missing values,
- tables and joins,
- entity IDs and primary-foreign keys,
- event logs and time order,
- treatment, outcome, covariate, mediator, and instrument roles.

### 7.2 Causal Query Encoder

Represents:

- treatment variable and intervention value,
- outcome and horizon,
- target population,
- estimand type,
- adjustment strategy or known graph constraints,
- cost, utility, and policy constraints when relevant.

### 7.3 Prior / Simulator

Generates training episodes:

- SCMs with varied graph structures,
- nonlinear mechanisms,
- hidden confounders,
- selection bias,
- missingness,
- measurement error,
- interference,
- time-varying treatments,
- randomized experiments,
- quasi-experiments,
- relational schemas,
- distribution shifts.

This is the heart of the approach. If the prior is too narrow, the model can be
fast and confident but wrong outside the prior.

### 7.4 Inference Heads

Possible heads:

- ATE/CATE posterior,
- graph posterior,
- adjustment set recommendation,
- counterfactual outcome generator,
- sensitivity bounds,
- policy value estimator,
- overlap and diagnostics classifier.

### 7.5 Guardrail Layer

Should report:

- what estimand was answered,
- which assumptions were used,
- whether identification was possible,
- whether overlap/positivity is weak,
- whether estimates are sensitive to hidden confounding,
- whether the query is out-of-prior or out-of-distribution.

## 8. Enterprise Use Cases Worth Studying

### Marketing and Growth

Questions:

- Who should receive an offer?
- Which campaign caused incremental conversion?
- What is the effect of contact frequency on retention?

Why causal:

High-propensity customers are not necessarily high-uplift customers. Prediction
targets the likely buyer; causal uplift targets the persuadable buyer.

### Pricing and Promotions

Questions:

- What happens if price changes for a segment?
- Which discount increases profit rather than just conversion?
- How do promotions cannibalize future demand?

Why causal:

Observed buyers at a price are selected. Price changes affect demand,
competition, inventory, and future expectations.

### Manufacturing and Operations

Questions:

- Which process parameter causes defect reduction?
- Which maintenance intervention prevents downtime?
- What is the root cause of a distribution shift?

Why causal:

Sensors are correlated through physical mechanisms. Root-cause analysis requires
distinguishing causes from downstream symptoms.

### Healthcare and Life Sciences

Questions:

- Which treatment benefits which patient subgroup?
- What is the effect of a protocol change?
- Which biomarker is causal versus merely predictive?

Why causal:

Treatment assignment is confounded by severity, access, clinician judgment, and
time-varying health states.

### Credit, Risk, and Policy

Questions:

- What is the effect of approving a borderline applicant?
- How would a policy change alter defaults and fairness metrics?
- Which intervention reduces delinquency?

Why causal:

Historical outcomes are produced by past policies. Predictive risk scores can
reinforce policy feedback loops.

### Recommendations and Marketplaces

Questions:

- What is the incremental effect of showing an item?
- How do seller interventions affect buyers and other sellers?
- Which ranking policy maximizes long-term welfare?

Why causal:

Exposure is controlled by the platform. Interference and feedback are central,
not edge cases.

### Supply Chain and Logistics

Questions:

- What is the effect of changing reorder thresholds?
- Which supplier intervention reduces delays?
- How do disruptions propagate through entities?

Why causal:

The system is relational, temporal, and intervention-heavy.

## 9. Research Questions for This Repo

### Definition

- What exact class of causal queries should the first model support?
- Is the target ATE/CATE, graph discovery, counterfactual generation, policy
  learning, or relational root-cause analysis?
- What assumptions are allowed: ignorability, known graph, partial graph,
  interventions, instruments, time order?

### Data Generation

- What SCM prior resembles enterprise tables?
- How should relational schemas be generated?
- How do we simulate realistic confounding, missingness, selection bias, and
  feedback?
- Can public business-like datasets be converted into semi-synthetic causal
  benchmarks?

### Model

- Should the first model be a PFN trained from scratch, a TabPFN-style adapter,
  or a pipeline around existing causal estimators?
- How should the model encode schema and query metadata?
- Should it output scalar effects, posterior samples, bounds, or diagnostics?

### Evaluation

- Does the model recover effects under known SCMs?
- Does uncertainty have correct coverage?
- Does performance degrade gracefully under hidden confounding?
- Does it know when the effect is not identifiable?
- Does it beat classical causal ML baselines under matched assumptions?
- Does it fail loudly under prior mismatch?

### Productization

- Can it plug into a DoWhy-like workflow?
- Can it produce an auditable causal report?
- Can a domain expert inspect assumptions before deployment?
- Can it recommend experiments or data collection to reduce uncertainty?

## 10. Suggested MVP Roadmap

### Phase 0: Build Baseline Literacy

Deliverables:

- A causal inference notebook using DoWhy or EconML.
- A simple SCM simulator.
- A small benchmark comparing naive regression, doubly robust estimation, and a
  PFN/tabular model used only as a nuisance model.

Datasets:

- IHDP,
- Lalonde / NSW,
- ACIC-style semi-synthetic datasets,
- one synthetic enterprise SCM such as pricing, churn, or maintenance.

### Phase 1: Causal Estimation Prototype

Goal:

Train or adapt a small PFN that estimates ATE/CATE from simulated SCM episodes
under a narrow assumption set.

Scope:

- binary treatment,
- tabular covariates,
- single outcome,
- no time-varying treatment,
- known ignorability in the first version,
- output effect estimate plus uncertainty.

Baselines:

- T-learner,
- S-learner,
- doubly robust learner,
- causal forest,
- Dragonnet/TARNet-style neural baselines,
- CausalPFN or Do-PFN if available.

### Phase 2: Assumption and Robustness Layer

Add:

- overlap diagnostics,
- balance diagnostics,
- placebo outcome tests,
- negative control hooks,
- hidden-confounding sensitivity,
- prior-mismatch tests.

Success criterion:

The system should say "not enough support" or "effect is assumption-sensitive"
when appropriate.

### Phase 3: Relational Causal Prior

Move from flat tables to multi-table schemas:

- customers-orders-products,
- patients-visits-treatments,
- machines-sensors-maintenance-events,
- suppliers-parts-shipments.

Key additions:

- entity resolution,
- temporal joins,
- leakage prevention,
- interference modeling,
- intervention propagation across relations.

### Phase 4: Experiment Recommendation

Use the model to suggest what intervention or randomized experiment would most
reduce uncertainty.

This is high leverage because, in causal inference, better data collection often
beats a more complex estimator.

## 11. Evaluation Checklist

Use this checklist for every experiment:

- Estimand is explicitly named.
- Unit of analysis is explicit.
- Treatment assignment process is stated.
- Outcome window is stated.
- Covariates are pre-treatment unless intentionally modeling mediation.
- Adjustment set is justified.
- Positivity/overlap is checked.
- Hidden confounding sensitivity is measured.
- Calibration or coverage is measured when uncertainty is reported.
- OOD/prior-mismatch tests are included.
- Classical causal baselines are included.
- Predictive metrics are not used as substitutes for causal validity.
- Reports distinguish "predictive correlation" from "intervention effect."

## 12. Common Failure Modes

**Mistaking prediction for intervention**

A model can predict who buys and still fail to estimate who will buy because of
an intervention.

**Automating away the estimand**

If the treatment, outcome, population, and time horizon are vague, the answer is
not actionable.

**Conditioning on post-treatment variables**

Including variables affected by treatment can block real effects or open biased
paths.

**Graph discovery overconfidence**

Many graphs can imply the same observational distribution. Discovery needs
assumptions, interventions, invariance, time, or prior knowledge.

**Prior mismatch**

PFNs inherit their simulated prior. A model trained on one family of SCMs can be
wrong on another while appearing confident.

**Unmeasured confounding**

No architecture removes the need to reason about missing common causes.

**Leaky relational features**

Feature engineering across time and tables can accidentally use future
information.

**Bad uncertainty**

Credible intervals from a prior-driven model may not have frequentist coverage
under real-world data-generating processes.

## 13. Reading Path

### Causal Inference Core

1. Learn potential outcomes, confounding, exchangeability, positivity, and
   consistency.
2. Learn SCMs, DAGs, interventions, back-door/front-door adjustment, and
   do-calculus.
3. Practice with DoWhy/EconML on a known dataset.

Suggested sources:

- Hernan and Robins, *Causal Inference: What If*
- Peters, Janzing, and Scholkopf, *Elements of Causal Inference*
- DoWhy paper and docs

### Foundation Model Core

1. Understand the foundation model concept.
2. Understand in-context learning.
3. Understand PFNs and TabPFN.
4. Understand tabular foundation model scaling limits.

Suggested sources:

- Bommasani et al., *On the Opportunities and Risks of Foundation Models*
- TabPFN original paper and TabPFN v2 Nature paper
- Statistical foundations of PFNs
- TabDPT and TabICL

### Causal Foundation Model Core

1. Read CInA for attention/covariate balancing intuition.
2. Read Do-PFN and CausalPFN for causal effect estimation.
3. Read CausalFM for the broader PFN causal inference framework.
4. Read consistency and simulation-stress-test papers before trusting
   uncertainty.
5. Read CausalTimePrior for temporal extensions.

## 15. Working Position for This Project

The most productive project stance is:

> Build toward causal foundation models for structured enterprise data by first
> treating them as assumption-aware, PFN-style amortized causal estimators, not as
> general causal oracles.

Near-term focus:

- tabular ATE/CATE under explicit assumptions,
- synthetic SCM prior design,
- uncertainty and sensitivity diagnostics,
- comparison against classical causal ML,
- roadmap toward relational and temporal causal data.

Long-term ambition:

- a causal foundation model that consumes enterprise schemas, time, experiments,
  observational data, and causal queries, then returns auditable intervention
  estimates with calibrated uncertainty and assumption diagnostics.
